# Kubernetes Events

If you've ever stared at a Pod stuck in `Pending` and wondered what Kubernetes is actually doing with it, or watched a container enter `CrashLoopBackOff` and wanted to understand the sequence of events leading up to that point, then Kubernetes Events are exactly the tool you need. Events are the cluster's activity log, a running commentary of everything significant that happens, generated by the same controllers and system components that make Kubernetes work.

:::info
Events are genuine Kubernetes objects with a kind, apiVersion, metadata, and spec, they live in etcd alongside Pods and Deployments, and you can list, describe, and filter them with `kubectl` just like any other resource.
:::

## What Are Events?

Every meaningful action in Kubernetes generates an Event object. Common triggers include:

- The scheduler placing a Pod on a node
- The kubelet pulling a container image
- A liveness probe failing
- A node running out of memory and evicting a Pod
- A Deployment scaling up its replica count

The major limitation of Events is their retention window. By default, Kubernetes only keeps Events for **one hour** after they were last observed. After that, they are garbage collected automatically. This means Events are great for understanding what's happening right now or in the very recent past, but they are not a historical record.

## The Most Common Way to Read Events

In practice, the most common way most Kubernetes users encounter Events is at the bottom of a `kubectl describe` output. When you run:

```bash
kubectl describe pod <pod-name>
```

The last section of the output is a table of Events associated with that specific Pod. This is the fastest way to understand what has happened to a Pod, from scheduling through image pulling through container startup.

```
Events:
  Type     Reason            Age   From               Message
  ----     ------            ---   ----                -------
  Normal   Scheduled         45s   default-scheduler  Successfully assigned default/my-pod to node-1
  Normal   Pulling           43s   kubelet            Pulling image "nginx:1.28"
  Normal   Pulled            38s   kubelet            Successfully pulled image "nginx:1.28"
  Normal   Created           38s   kubelet            Created container nginx
  Normal   Started           37s   kubelet            Started container nginx
```

This tells the complete story of the Pod's journey: it was scheduled, the image was pulled, the container was created, and it started, all in the span of about 8 seconds.

## Listing Events Directly

You can also query Events directly as first-class objects. To list all Events in the current namespace:

```bash
kubectl get events
```

Expected output (abbreviated):

```
LAST SEEN   TYPE      REASON              OBJECT              MESSAGE
2m          Normal    Scheduled           Pod/my-pod          Successfully assigned default/my-pod to node-1
2m          Normal    Pulled              Pod/my-pod          Container image already present on machine
2m          Normal    Created             Pod/my-pod          Created container nginx
2m          Warning   BackOff             Pod/crasher-pod     Back-off restarting failed container
5m          Normal    ScalingReplicaSet   Deployment/api      Scaled up replica set api-7d9b4f8b to 3
```

By default, `kubectl get events` sorts events by the time they were first observed, which may not match the order you expect. To ensure the output is in strict chronological order, sort by `lastTimestamp`:

```bash
kubectl get events --sort-by=.lastTimestamp
```

To see Events in a specific namespace, or across all namespaces:

```bash
kubectl get events -n kube-system
kubectl get events -A
```

## Event Fields in Detail

Each Event contains several fields that help you understand what happened and why:

**`Type`** is either `Normal` or `Warning`. Normal events are purely informational, "image pulled", "container started", "Pod scheduled". Warning events indicate something that may require attention: a probe failure, a scheduling constraint that couldn't be satisfied, an OOMKill, an image pull failure.

**`Reason`** is a short, machine-readable code describing the cause. Examples: `Scheduled`, `Pulling`, `Pulled`, `Created`, `Started`, `BackOff`, `Failed`, `OOMKilling`, `FailedScheduling`, `EvictionThresholdMet`. These reasons are consistent across Kubernetes versions and are useful for scripting and filtering.

**`Object`** identifies which Kubernetes resource the Event is about, for example, `Pod/my-pod` or `Deployment/my-deployment`.

**`Message`** is a human-readable explanation of what happened. This is often the most useful field for debugging because it contains the actual error text, the specific constraint that failed, or the image name that couldn't be pulled.

**`Source`** identifies which component generated the Event. Common sources include `default-scheduler` (for scheduling decisions), `kubelet` (for container lifecycle events), `replicaset-controller` (for ReplicaSet changes), and `horizontal-pod-autoscaler`.

**`Count`** and the related `FirstTime`/`LastTime` fields tell you how many times an Event has been observed and over what time span. When a container is in a crash loop, the `BackOff` Event count will grow rapidly. Seeing "Count: 47" on a `BackOff` event tells you the container has been restarting frequently.

## Filtering Events with Field Selectors

You can filter Events to show only those related to a specific object using `--field-selector`:

```bash
kubectl get events --field-selector involvedObject.name=my-pod
```

Or filter by Event type to focus on warnings only:

```bash
kubectl get events --field-selector type=Warning
```

You can combine multiple field selectors:

```bash
kubectl get events --field-selector type=Warning,involvedObject.kind=Pod
```

## Events for System Components

System components in the `kube-system` namespace also generate Events. When you're debugging a node that won't accept new Pods, or a networking issue involving CoreDNS, or a persistent volume that won't bind, checking the `kube-system` Events is often informative:

```bash
kubectl get events -n kube-system --sort-by=.lastTimestamp
```

## The 1-Hour Retention Problem

The one-hour Event retention window is often too short for serious post-incident analysis. If an incident happened at 2 AM and your on-call engineer wakes up at 6 AM, all the Events from the incident are gone. This is a significant limitation.

The solution is to export Events to a persistent external system. Tools like **kube-event-exporter** and **Eventrouter** (from Heptio/VMware) watch for Events in real time and forward them to Elasticsearch, Loki, Kafka, or other backends where they can be stored indefinitely and queried after the fact.

:::info
Some platforms and managed Kubernetes services extend the Event retention window (Google GKE, for example, provides longer retention for audit-related Events). Check your platform's documentation to understand exactly what is available to you.
:::

## Hands-On Practice

**Step 1: Create a Pod and watch its Events**

```yaml
# event-demo-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: event-demo
spec:
  containers:
    - name: nginx
      image: nginx:1.28
```

```bash
kubectl apply -f event-demo-pod.yaml
```

Wait a moment, then describe the Pod and read the Events section:

```bash
kubectl describe pod event-demo
```

Look at the bottom of the output. You should see Events for scheduling, image pulling, and container start.

**Step 2: List all Events sorted by time**

```bash
kubectl get events --sort-by=.lastTimestamp
```

**Step 3: Create a Pod with a bad image and observe Warning Events**

```yaml
# bad-image-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: bad-image-pod
spec:
  containers:
    - name: app
      image: nginx:this-tag-does-not-exist
```

```bash
kubectl apply -f bad-image-pod.yaml
```

Wait 20 seconds, then:

```bash
kubectl get events --field-selector involvedObject.name=bad-image-pod
```

Expected output:

```
LAST SEEN   TYPE      REASON          OBJECT              MESSAGE
15s         Normal    Scheduled       Pod/bad-image-pod   Successfully assigned default/bad-image-pod to node-1
13s         Normal    Pulling         Pod/bad-image-pod   Pulling image "nginx:this-tag-does-not-exist"
10s         Warning   Failed          Pod/bad-image-pod   Failed to pull image "nginx:this-tag-does-not-exist": ...
10s         Warning   Failed          Pod/bad-image-pod   Error: ErrImagePull
5s          Warning   BackOff         Pod/bad-image-pod   Back-off pulling image "nginx:this-tag-does-not-exist"
```

Notice the `Type` column, `Warning` events appear for the failed pull and the subsequent back-off. This is exactly the pattern you'll see for any image-related failure in the wild.

**Step 4: Filter for only Warning events**

```bash
kubectl get events --field-selector type=Warning
```

**Step 5: Check kube-system events**

```bash
kubectl get events -n kube-system --sort-by=.lastTimestamp | tail -10
```

**Step 6: Clean up**

```bash
kubectl delete pod event-demo bad-image-pod
```
